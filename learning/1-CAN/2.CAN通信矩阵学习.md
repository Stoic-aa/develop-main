# 基本概念

## 1.1 CAN 通信矩阵解释
CAN 通信矩阵解释 指的是 CAN 控制器的局域网络通信中, 总线系统中的所有通信报文及其包含信号的**结构化, 标准化**的数据规范文档,**通常的体现就是我们的 .dbc, 数据库文件, Excel表格或专用的格式**, 通常由OEM或系统集成者进行制定和主导设计, 定义并发布, 是整车电子电气架构中各电子控制单元(ECU)之间通过 CAN总线进行数据交互的唯一权威契约和通信蓝图

## 1.2 基本术语
1. Eetwork-CAN : CAN网络(理解为电话簿)
2. Node-CAM    : CAN节点(域控制器/ECU)(理解为联系人)
3. Message     : CAN数据/报文(理解为聊天话题)
4. Signal      : CAN信号(理解为聊条的内容)

## 1.3 属性

1. message 属性:
    - 常用的是: Name, Type, ID, DLC, Transmitter(发送方), Tx Method(Cycle, Event, CycleEvent), Cycle Time

2. signal 属性:
    - 常用的属性是:Name, length(bit), Byte Order(字节序), Value Type(值类型), Unit(单位), Factor(精度系统), offet(偏移量), Min/Max, Value Table(值表)

**Note**

1. **physical signal**:physical signal = Raw data * Factor + offset

2. Byte Order 字节序:**不代表传输顺序**
    - 一般有intel模式和motorola模式
    - intel模式    : intel 采用的是小端(0XAABB存储为 BB AA这种形式)
    - motorola模式 : 采用的是大端
3. 总结
    - 不管 intel 模式还是 motorola 模式, 起始位都是 LSB
    - 信号在不夸字节的情况下, intel 和 motorola 格式编码结果是没有区别的
    - 一个 DBC 数据库中信号的排布方式只能选择一种, 一般不会出现两者共同存在一个 DBC 数据库中
